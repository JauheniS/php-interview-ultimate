# 7. MySQL и базы данных

### Как подключиться к базе данных MySQL из PHP?

<span className="badge badge--success margin-bottom--md">Junior</span>

**Ответ:** Можно использовать расширение `mysqli` или `PDO` (PHP Data Objects).

```php
// Пример с PDO
$pdo = new PDO('mysql:host=localhost;dbname=test', 'user', 'pass');
```

---

### В чём разница между `mysqli` и `PDO`?

<span className="badge badge--success margin-bottom--md">Junior</span>

**Ответ:**

- `mysqli` работает только с MySQL, тогда как `PDO` поддерживает несколько систем управления базами данных (PostgreSQL, SQLite и др.).
- `PDO` поддерживает именованные плейсхолдеры в подготовленных выражениях.

---

### В чём разница между `WHERE` и `HAVING`?

<span className="badge badge--success margin-bottom--md">Junior</span>

**Ответ:** `WHERE` фильтрует строки _до_ агрегации, а `HAVING` фильтрует _после_ агрегации (обычно совместно с `GROUP BY`).

---

### Что такое Primary Key?

<span className="badge badge--success margin-bottom--md">Junior</span>

**Ответ:** Первичный ключ — это столбец (или набор столбцов), который однозначно идентифицирует каждую запись в таблице. Он не может быть null.

---

### В чём разница между `CHAR` и `VARCHAR`?

<span className="badge badge--success margin-bottom--md">Junior</span>

**Ответ:** `CHAR` имеет фиксированную длину (дополняется пробелами), тогда как `VARCHAR` имеет переменную длину (использует только необходимое пространство плюс 1–2 байта на длину).

---

### Какой порт по умолчанию у MySQL?

<span className="badge badge--success margin-bottom--md">Junior</span>

**Ответ:** 3306.

---

### В чём разница между `CHAR_LENGTH` и `LENGTH`?

<span className="badge badge--success margin-bottom--md">Junior</span>

**Ответ:** `CHAR_LENGTH` считает количество символов, а `LENGTH` — количество байтов.

---

### Что означают `%` и `_` в выражении `LIKE`?

<span className="badge badge--success margin-bottom--md">Junior</span>

**Ответ:**

- `%` соответствует любой последовательности символов (ноль или более).
- `_` соответствует ровно одному символу.

---

### Что такое транзакция?

<span className="badge badge--primary margin-bottom--md">Middle</span>

**Ответ:** Последовательность операций с базой данных, которые рассматриваются как единая единица работы. Они должны соответствовать свойствам ACID.

---

### Какие основные движки хранения данных есть в MySQL?

<span className="badge badge--primary margin-bottom--md">Middle</span>

**Ответ:**

- **InnoDB:** Поддерживает транзакции, блокировку на уровне строк и внешние ключи (используется по умолчанию).
- **MyISAM:** Не поддерживает транзакции, использует блокировку на уровне таблиц, быстрее при нагрузке с преобладанием чтения.
- **MEMORY:** Хранит данные в оперативной памяти для очень быстрого доступа, но данные теряются при перезапуске сервера.

---

### Что такое представление (View)?

<span className="badge badge--primary margin-bottom--md">Middle</span>

**Ответ:** Виртуальная таблица, основанная на результате SQL-запроса. Она не хранит данные самостоятельно.

---

### Что такое триггер (Trigger)?

<span className="badge badge--primary margin-bottom--md">Middle</span>

**Ответ:** Объект базы данных, который автоматически выполняется в ответ на определённые события (INSERT, UPDATE, DELETE) в конкретной таблице.

---

### Что такое InnoDB?

<span className="badge badge--primary margin-bottom--md">Middle</span>

**Ответ:** Движок хранения данных для MySQL, поддерживающий транзакции, внешние ключи и блокировку на уровне строк. Является движком по умолчанию.

---

### Что такое индекс и зачем он нужен?

<span className="badge badge--primary margin-bottom--md">Middle</span>

**Ответ:** Индекс — это структура данных, используемая для быстрого поиска записей в таблице. Он ускоряет выполнение запросов, но может замедлять вставку и обновление данных.

---

### Как хранятся индексы?

<span className="badge badge--primary margin-bottom--md">Middle</span>

**Ответ:** Большинство индексов баз данных (включая InnoDB в MySQL) хранятся в виде **B-деревьев** (точнее, B+Trees). Они «горизонтально широкие» (высокий коэффициент ветвления) и неглубокие, что минимизирует количество операций дискового ввода-вывода для поиска значения. Такая структура обеспечивает быструю фильтрацию (O(log n)) и эффективное сканирование диапазонов. Правильный порядок столбцов в составных индексах критически важен: как правило, столбцы с условиями равенства и высокой селективностью должны идти первыми, за ними — столбцы с условиями диапазона.

---

### В чём разница между `UNION` и `UNION ALL`?

<span className="badge badge--primary margin-bottom--md">Middle</span>

**Ответ:** `UNION` удаляет дублирующиеся строки из результата, тогда как `UNION ALL` включает все строки, в том числе дубликаты.

---

### В чём разница между MariaDB и MySQL?

<span className="badge badge--primary margin-bottom--md">Middle</span>

**Ответ:** MariaDB — это форк MySQL с открытым исходным кодом, созданный оригинальными разработчиками после приобретения MySQL компанией Oracle. Сохраняя высокую совместимость, MariaDB предлагает больше движков хранения, улучшения производительности и остаётся полностью открытой (GPL), тогда как MySQL имеет как community-, так и enterprise-версию.
[Сравнение MariaDB и MySQL](/answers/mysql_advanced#9-mariadb-vs-mysql)

---

### Что такое шардирование и партиционирование?

<span className="badge badge--warning margin-bottom--md">Senior</span>

**Ответ:**

- **Партиционирование:** Разделение большой таблицы на более мелкие, удобные для управления части в рамках одного сервера базы данных.
- **Шардирование:** Распределение данных между несколькими независимыми серверами баз данных (горизонтальное масштабирование).

---

### Что такое свойства ACID?

<span className="badge badge--warning margin-bottom--md">Senior</span>

**Ответ:**

- **Атомарность (Atomicity):** Всё или ничего (либо все операции выполняются успешно, либо ни одна).
- **Согласованность (Consistency):** Транзакции должны переводить базу данных из одного допустимого состояния в другое.
- **Изолированность (Isolation):** Транзакции выполняются независимо друг от друга.
- **Долговечность (Durability):** Зафиксированные данные сохраняются навсегда, даже при сбое системы.

---

### В чём разница между кластеризованным и некластеризованным индексом?

<span className="badge badge--warning margin-bottom--md">Senior</span>

**Ответ:**

- **Кластеризованный индекс:** Определяет физический порядок данных в таблице. В таблице может быть только один (обычно это первичный ключ).
- **Некластеризованный индекс:** Отдельная структура, содержащая индексированные значения и указатели на строки данных.

---

### Какие уровни изоляции вы знаете?

<span className="badge badge--warning margin-bottom--md">Senior</span>

**Ответ:** Стандартные уровни изоляции: Read Uncommitted, Read Committed, Repeatable Read и Serializable. Они определяют баланс между согласованностью и производительностью при параллельных транзакциях.
[Подробное руководство по уровням изоляции](/answers/mysql_advanced#5-уровни-изоляции-транзакций)

---

### В чём разница между Read Committed и «грязным чтением» (Dirty Read)?

<span className="badge badge--warning margin-bottom--md">Senior</span>

**Ответ:** «Грязное чтение» (Dirty Read) возникает на уровне Read Uncommitted, когда транзакция видит незафиксированные данные другой транзакции. Read Committed предотвращает это, гарантируя, что транзакция видит только уже зафиксированные данные.
[Подробное руководство по уровням изоляции](/answers/mysql_advanced#5-уровни-изоляции-транзакций)

---

### Приведите примеры уровней изоляции транзакций и проблем, которые они решают.

<span className="badge badge--warning margin-bottom--md">Senior</span>

**Ответ:**

- **Read Uncommitted:** Чтение баланса банковского счёта, который обновляется, но ещё не зафиксирован (грязное чтение).
- **Read Committed:** Отчёт дважды читает одни и те же данные и получает разные результаты, потому что другая транзакция зафиксировала обновление между чтениями (неповторяемое чтение).
- **Repeatable Read:** Обеспечение согласованного снимка данных на протяжении долгой транзакции, даже если другие транзакции фиксируют изменения.
- **Serializable:** Финансовые системы, где требуется абсолютная корректность, предотвращение всех проблем параллелизма, включая фантомное чтение, ценой производительности.
  [Подробное руководство по уровням изоляции](/answers/mysql_advanced#51-практические-примеры-уровней-изоляции)

### В чём разница между MySQL и PostgreSQL?

<span className="badge badge--warning margin-bottom--md">Senior</span>

**Ответ:** MySQL известен своей скоростью и простотой использования в веб-разработке, тогда как PostgreSQL — это объектно-реляционная база данных, известная своими расширенными возможностями, строгим соответствием стандарту SQL и лучшей обработкой сложных запросов и высоконагруженных параллельных сценариев.

### Есть фильтр по 3 столбцам одной таблицы: `sku` (уникальный), `creation_date` и `integer_field` (значения 1, 2 или 3). Как проиндексировать для максимальной производительности?

<span className="badge badge--warning margin-bottom--md">Senior</span>

**Ответ:** Создать **составной индекс**, включающий все три поля. Для максимальной производительности порядок должен быть: **sku** (наибольшая селективность), затем **integer_field** (точные совпадения), и наконец **creation_date** (фильтр по диапазону). Такой порядок обеспечивает немедленное сужение результатов по самому селективному фильтру, а фильтр по диапазону идёт последним, чтобы предшествующие части индекса использовались полностью.
[Стратегия составных индексов](/answers/mysql_advanced#11-стратегия-составных-индексов)

### У вас есть такой SQL-запрос. Что в нём не так и как его улучшить?

`SELECT * FROM table ORDER BY id LIMIT :limit OFFSET :offset`

- `id` — это Primary Key, AutoIncrement, с индексом.

<span className="badge badge--warning margin-bottom--md">Senior</span>

**Ответ:**

Использование `OFFSET` для пагинации неэффективно для больших объёмов данных, поскольку MySQL должен просканировать и отбросить все строки до смещения. Например, `OFFSET 1000000` требует прочитать 1 миллион строк, прежде чем вернуть запрошенные, что ведёт к высокой нагрузке на ввод-вывод и процессор.

**Оптимизация (курсорная пагинация):**

- **Начальный запрос:** `SELECT * FROM table ORDER BY id LIMIT :limit`
- **Последующие запросы:** Берём **последний ID** с текущей страницы (например, `:last_id`) и используем его для пропуска предыдущих строк:
  `SELECT * FROM table WHERE id > :last_id ORDER BY id LIMIT :limit`

Это гарантирует, что запрос остаётся быстрым вне зависимости от номера страницы, поскольку используется индекс первичного ключа для перехода непосредственно к нужной точке.

### У вас есть такие SQL-запросы, какие индексы вы бы использовали?

1. `SELECT * FROM transactions WHERE account_id = :account_id AND transaction_type = :transaction_type AND created_at BETWEEN :start AND :end`
2. `SELECT * FROM transactions WHERE account_id = :account_id AND transaction_type = :transaction_type ORDER BY created_at`

- `account_id`: bigint, ~100 000 вариантов
- `transaction_type`: int, 15 вариантов
- `created_at`: DATETIME, DEFAULT NOW()

<span className="badge badge--warning margin-bottom--md">Senior</span>

**Ответ:**

Использовать составной индекс: `idx (account_id, transaction_type)`.

**Обоснование:**

- Лучше сначала фильтровать по `account_id`, чтобы значительно сузить область поиска (высокая селективность).
- Затем фильтровать по `transaction_type`.
- Во втором запросе `created_at` используется для сортировки. Хотя добавление его в индекс может избежать `filesort`, это не всегда целесообразно, если столбец не сильно сужает выборку в большинстве запросов, или если отфильтрованный набор данных достаточно мал для эффективной сортировки после выборки.

### У вас есть такой код, что в нём не так?

```php
public function addBalance(float $sum, User $user) {
    $balance = $this->repo->getUserBalance($user->getId());
    $balance += $sum;
    $this->repo->updateUserBalance($balance, $user->getId());
}
// В getUserBalance: SELECT balance FROM Users WHERE id = :id
// В updateUserBalance: UPDATE Users SET balance = :balance WHERE id = :id
```

<span className="badge badge--warning margin-bottom--md">Senior</span>

**Ответ:**

Этот код уязвим к **состоянию гонки** (потерянные обновления). Если два параллельных процесса прочитают один и тот же баланс до того, как любой из них выполнит обновление, одно обновление перезапишет другое.

**Подробное объяснение:**

- На самом слабом уровне изоляции (**Read Uncommitted**) может произойти **грязное чтение** (чтение незафиксированных данных).
- Даже при уровнях изоляции по умолчанию параллельные циклы «чтение — изменение — запись» приводят к потере данных.
- **Решение:** Использовать **`SELECT ... FOR UPDATE`** в MySQL. Это блокирует конкретную строку, заставляя другие транзакции ждать освобождения блокировки (после фиксации `UPDATE`).

**Правильная реализация:**

```php
$this->db->transaction(function() use ($sum, $user) {
    // SELECT balance FROM Users WHERE id = :id FOR UPDATE
    $balance = $this->repo->getUserBalanceForUpdate($user->getId());
    $balance += $sum;
    $this->repo->updateUserBalance($balance, $user->getId());
});
```

### Как обеспечить идемпотентность при вставке записей (например, в консьюмере брокера сообщений)?

<span className="badge badge--warning margin-bottom--md">Senior</span>

**Ответ:** Используйте **Unique Constraints** на соответствующих бизнес-ключах (например, `(order_id, user_id)`). При повторной доставке сообщения (гарантия "at-least-once") база данных заблокирует вторую вставку. Приложение должно перехватывать ошибку `Duplicate Key` и обрабатывать её как успех или известное состояние, чтобы обеспечить корректный результат для клиента.

### Как управлять лимитами (например, максимальным количеством участников) в условиях высокой конкуренции?

<span className="badge badge--warning margin-bottom--md">Senior</span>

**Ответ:**
1.  **Пессимистическая блокировка**: `SELECT * FROM meetups WHERE id = 1 FOR UPDATE;` для сериализации всех регистраций на этот конкретный митап.
2.  **Денормализация + CHECK Constraint**: Добавьте колонку `booked_seats` и ограничение `CHECK (booked_seats <= total_seats)`. Обновляйте счётчик в той же транзакции, что и вставку посетителя. Это позволяет БД самой контролировать соблюдение лимита.
[Подробное руководство по конкурентности и лимитам в SQL](../answers/sql_concurrency.md)

### Что такое `xmin` и `xmax` в контексте строк базы данных (например, в PostgreSQL)?

<span className="badge badge--warning margin-bottom--md">Senior</span>

**Ответ:** Это скрытые системные колонки, используемые для реализации **MVCC (Multi-Version Concurrency Control)**.
- **PostgreSQL**:
  - `xmin`: хранит ID транзакции, которая вставила строку.
  - `xmax`: хранит ID транзакции, которая удалила или обновила строку.
- **MySQL (InnoDB)**:
  - `DB_TRX_ID`: аналогично `xmin`, хранит ID транзакции, которая создала версию строки.
  - `DB_ROLL_PTR`: указатель на предыдущую версию в **Undo Log** (используется для восстановления старых версий).
  - `DB_ROW_ID`: скрытый ID строки (если нет первичного ключа).
Понимание этих колонок помогает объяснить, как разные транзакции «видят» разные версии данных одновременно без блокировки друг друга.

### Когда следует использовать блокировки Redis вместо встроенных возможностей SQL?

<span className="badge badge--warning margin-bottom--md">Senior</span>

**Ответ:** Используйте Redis для **Rate Limiting** (ограничения частоты запросов), **Circuit Breakers** (предохранителей) или **синхронизации задач между узлами** (например, запуск Cron-задачи один раз). Избегайте использования Redis для обеспечения консистентности данных *внутри* базы данных. Функции SQL (уникальные индексы, ограничения, блокировки строк) более надёжны, лучше справляются со сбоями (нет проблем с TTL) и не создают лишних точек отказа или накладных расходов на уровне данных.
