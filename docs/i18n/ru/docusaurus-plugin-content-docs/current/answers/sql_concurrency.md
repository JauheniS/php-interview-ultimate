# SQL-грамотность: изоляция, блокировки и идемпотентность

Это руководство охватывает типичные сценарии конкурентного доступа к базе данных на основе практического опыта. Понимание этих концепций критически важно для построения надежных и высоконагруженных бэкенд-приложений.

---

## 1. Кейс №1: Состояние гонки при обновлении (Update Race Condition)

### Сценарий
Система бронирования слотов для митапов. Несколько пользователей могут одновременно нажать кнопку «Записаться».

**Наивная реализация:**
```php
$slot = $db->query("SELECT * FROM meetup_slots WHERE id = 1")->fetch();
if ($slot->speaker_id === null) {
    $db->query("UPDATE meetup_slots SET speaker_id = $userId WHERE id = 1");
}
```

**Проблема:** два пользователя одновременно видят `speaker_id IS NULL`. Оба выполняют `UPDATE`. Второй пользователь перезаписывает первого.

### Решения:
1.  **Пессимистическая блокировка (`SELECT FOR UPDATE`)**:
    Блокирует строку до завершения транзакции. Другие транзакции будут ждать на этапе `SELECT`. Используйте **`SKIP LOCKED`**, если хотите пропустить уже заблокированные строки (полезно для реализации очередей).
2.  **Атомарные обновления (рекомендуется)**:
    Объедините проверку и обновление в одном SQL-запросе:
    ```sql
    UPDATE meetup_slots 
    SET speaker_id = :user_id 
    WHERE id = :id AND speaker_id IS NULL;
    ```
    Всегда проверяйте количество **Rows Affected**. Если оно равно `0`, значит слот уже занят.

## 2. Кейс №2: Идемпотентность (обработка повторов)

### Сценарий
API для добавления посетителя на митап. Из-за сетевых сбоев или гарантии доставки "at-least-once" в Kafka один и тот же запрос может прийти несколько раз.

**Проблема:** без защиты один и тот же пользователь может быть добавлен в список посетителей несколько раз, занимая лишние места.

### Решение:
1.  **Unique Constraints**: создайте уникальный индекс на пару `(meetup_id, user_id)`.
2.  **Обработка ошибок**:
    Перехватывайте ошибку `Duplicate Key` в коде. При повторном запросе база заблокирует вторую вставку. Ваше приложение должно обработать это как успех (или известное состояние), чтобы клиент получил корректный результат.

## 3. Кейс №3: Управление лимитами (агрегаты)

### Сценарий
На митапе ограниченное количество мест (например, 100). Нужно гарантировать, что мы его не превысим.

**Наивная реализация:**
```sql
SELECT COUNT(*) FROM visitors WHERE meetup_id = 1; -- Возвращает 99
-- ... если count < limit, то ...
INSERT INTO visitors (meetup_id, user_id) VALUES (1, 101);
```

**Проблема:** при высокой нагрузке две параллельные транзакции могут увидеть `99` и обе выполнят `INSERT`, превысив лимит (станет `101`).

### Решения:
1.  **Блокировка родительской строки**:
    `SELECT * FROM meetups WHERE id = 1 FOR UPDATE;`
    Это выстраивает все регистрации на *конкретный митап* в очередь.
2.  **Денормализация + CHECK Constraint**:
    Добавьте колонку `booked_seats` в таблицу `meetups` и `CHECK (booked_seats <= total_seats)`.
    Обновляйте счетчик в той же транзакции, что и вставку посетителя:
    ```sql
    UPDATE meetups SET booked_seats = booked_seats + 1 WHERE id = 1;
    ```
    База данных обеспечит выполнение `CHECK` для всех транзакций.

## 4. Распределенные блокировки (Redis vs SQL)

### Когда использовать Redis:
-   **Rate Limiting**: предотвращение шквала запросов к базе данных.
-   **Circuit Breakers**: временная блокировка доступа к ресурсу, если БД перегружена.
-   **Блокировка задач (Cron)**: гарантия того, что одна и та же задача не запустится одновременно на нескольких узлах.

### Когда избегать Redis для консистентности данных:
Если консистентность нужна *внутри* БД (например, защита от двойного бронирования), используйте возможности SQL (Unique Indexes, Constraints, Row Locks).
-   **Сложность**: Redis — это дополнительная точка отказа.
-   **Нет кросс-системной атомарности**: нельзя атомарно обновить ключ в Redis и строку в SQL в одной транзакции.
-   **Риски TTL**: блокировка в Redis может истечь раньше, чем завершится медленная SQL-транзакция.

## 5. Чек-лист для разработчика

- [ ] **Rows Affected**: всегда проверяйте `rows_affected` после `UPDATE` или `DELETE`.
- [ ] **Ошибки SQL**: явно обрабатывайте нарушение ограничений (Constraint Violation), взаимоблокировки (Deadlocks) и ошибки сериализации.
- [ ] **Unique Indexes**: используйте их для всех бизнес-критичных полей, а не только для первичного ключа.
- [ ] **Короткие транзакции**: держите блокировки как можно меньше времени.
- [ ] **Идемпотентность**: убедитесь, что повторный вызов операции не вызывает побочных эффектов или фатальных ошибок.

## 6. Глубокое погружение: знания уровня Expert

### MVCC и системные колонки
Современные базы данных используют **Multi-Version Concurrency Control (MVCC)** для управления конкурентным доступом без блокировки каждого чтения. В PostgreSQL каждая строка имеет скрытые системные колонки:
- **`xmin`**: ID транзакции, которая вставила строку.
- **`xmax`**: ID транзакции, которая удалила или обновила строку (изначально `0`).
Понимание этих механизмов помогает объяснить, как база данных «видит» разные версии данных на разных уровнях изоляции.

### Advisory Locks (Рекомендательные блокировки)
Иногда нужно заблокировать «логический» ресурс, который не соответствует конкретной строке в базе данных.
- **PostgreSQL**: `SELECT pg_advisory_lock(123);` (Блокировки на уровне сессии или транзакции).
- **MySQL**: `SELECT GET_LOCK('my_resource', 10);` (Глобальные блокировки с таймаутом).
**Предупреждение**: используйте их с осторожностью. Обычно лучше заблокировать реальную строку в таблице.

### Хранимые процедуры vs Логика приложения
Воркшоп поднимает классический спор:
- **Все в БД**: использование триггеров или хранимых процедур гарантирует консистентность, даже если к БД обращаются разные приложения.
- **Все в коде (рекомендуется)**: бизнес-логику в БД сложно версионировать, тестировать и легко забыть при миграциях. Рекомендуемый подход — использовать БД для **обеспечения схемы** (типы, Unique Index, CHECK) и держать бизнес-правила на уровне приложения.

## 7. Чек-лист для System Design

При проектировании функционала, связанного с общими ресурсами или лимитами:
1.  **Конкурентность**: есть ли одновременный доступ к одной и той же сущности от разных пользователей?
2.  **Фрод/Абуз**: может ли пользователь попытаться «обмануть» систему, нажав кнопку с нескольких устройств одновременно?
3.  **Уровень консистентности**: нужна ли вам **Strong Consistency** (например, для денег или лимитов) или допустима **Eventual Consistency**?
4.  **Идемпотентность**: является ли ваш API идемпотентным? Что произойдет, если запрос повторится после таймаута сети?
5.  **Режимы отказа**: что произойдет, если соединение оборвется *после* коммита в БД, но *до* того, как клиент получит ответ?
