# Проектирование и архитектура ПО

## Основы объектно-ориентированного программирования (ООП)

### Инвариантность, ковариантность и контравариантность
Эти термины описывают, как соотношение подтипов между сложными типами (такими как типы возвращаемых значений или типы параметров) связано с соотношением подтипов их составляющих.

- **Инвариантность**: тип должен совпадать точно.
- **Ковариантность**: позволяет дочернему классу возвращать *более конкретный* тип, чем родительский (например, `parent::make(): Animal` может быть переопределён как `child::make(): Dog`). Поддерживается в PHP 7.4+ для типов возвращаемых значений.
- **Контравариантность**: позволяет дочернему классу принимать *менее конкретный* (более широкий) тип, чем родительский (например, `parent::eat(Dog)` может быть переопределён как `child::eat(Animal)`). Поддерживается в PHP 7.4+ для типов параметров.

### Композиция vs Наследование
- **Наследование** (отношение «является»): используется, когда класс является специализированной версией другого. Риск: проблема хрупкого базового класса (Fragile Base Class), жёсткая связанность.
- **Композиция** (отношение «содержит»): используется, когда класс использует другой класс для выполнения задачи. Преимущество: более гибкая, легче изменить в рантайме, слабая связанность.
- **Правило**: «Предпочитайте композицию наследованию.»

### Почему геттеры и сеттеры считаются «плохой практикой» (иногда)
Их чрезмерное использование может привести к **анемичным доменным моделям**, где объекты представляют собой просто контейнеры данных (DSP — Data Structure Projections) без поведения. Это нарушает **инкапсуляцию**, поскольку вы раскрываете внутреннее состояние объекта.
- **Лучший подход**: «Говори, не спрашивай» (Tell, Don't Ask). Вместо того чтобы получить значение, выполнить логику и установить его обратно, скажите объекту выполнить действие самостоятельно.

## Service Locator vs Inversion of Control (IoC) Container

- **Service Locator**: паттерн, при котором объект «обращается» к центральному реестру для поиска своих зависимостей.
  - *Минусы*: скрытые зависимости, сложнее тестировать, нарушает принцип Dependency Inversion.
- **IoC Container (Dependency Injection)**: зависимости «внедряются» в объект (через конструктор или сеттер). Объект пассивен и не знает, как создаются его зависимости.
  - *Плюсы*: явные зависимости, проще мокировать/тестировать, слабо связанный код.

## GRASP (General Responsibility Assignment Software Patterns)
Набор из 9 паттернов/принципов для распределения ответственности между классами.
- **Information Expert**: назначьте ответственность классу, который обладает информацией, необходимой для её выполнения.
- **Creator**: кто должен отвечать за создание нового экземпляра класса?
- **Controller**: кто должен обрабатывать системное событие?
- **Low Coupling**: поддерживайте классы максимально независимыми друг от друга.
- **High Cohesion**: сохраняйте связанную логику в одном классе.
- **Indirection**: используйте промежуточный объект для развязывания двух компонентов.
- **Polymorphism**: обрабатывайте альтернативы на основе типа.
- **Pure Fabrication**: создайте класс, который не представляет концепцию предметной области, для достижения слабой связанности/высокой связности (например, Service).
- **Protected Variations**: защитите элементы от изменений других элементов, обернув источник нестабильности интерфейсом.

## Архитектурные принципы

### Закон Деметры (принцип наименьшего знания)
«Общайтесь только с непосредственными друзьями.» Объект не должен «проникать» через другой объект к третьему (например, избегайте `$user->getAccount()->getTransactions()->getFirst()`).

### Анемичная доменная модель
Доменная модель, в которой объекты имеют состояние (свойства), но мало логики (поведения) или не имеют её вовсе. Логика при этом находится в классах-«сервисах». Это часто считается анти-паттерном в DDD.

### DDD (Domain-Driven Design)
Подход к разработке ПО, в котором приоритет отдаётся предметной области и доменной логике.
- **Entity**: объект с уникальной идентичностью, сохраняющейся во времени (например, User).
- **Value Object**: объект, определяемый своими атрибутами, без идентичности (например, Address, Money). Должен быть неизменяемым (immutable).
- **DTO (Data Transfer Object)**: простой объект для передачи данных между слоями (без логики).

## Hexagonal (Ports & Adapters) vs Onion Architecture
Оба подхода направлены на отделение ядра приложения (бизнес-логики) от внешних зависимостей (БД, API, UI).
- **Application Core**: содержит доменную логику и не зависит от внешних инструментов.
- **Ports**: интерфейсы, определённые ядром для описания того, что ему нужно (например, `UserRepositoryInterface`).
- **Adapters**: конкретные реализации этих интерфейсов (например, `DoctrineUserRepository`).
- **Onion**: акцент на слоях, где «зависимости направлены внутрь». Самый внутренний слой — Domain Model.
