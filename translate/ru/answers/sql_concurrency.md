# SQL Literacy: Изоляция, Блокировки и Идемпотентность

Создание надежных и высоконагруженных бэкенд-приложений требует глубокого понимания того, как базы данных обрабатывают конкурентный доступ. Это руководство рассматривает типичные сценарии, подводные камни и экспертные стратегии обеспечения целостности данных и производительности.

---

## 1. Кейс №1: Состояние гонки при обновлении (Update Race Condition)

### Сценарий
Система, где несколько пользователей могут одновременно изменять один и тот же ресурс, например, бронировать «слот» на митапе.

**Наивная реализация:**
```php
$slot = $db->query("SELECT * FROM meetup_slots WHERE id = 1")->fetch();
if ($slot->speaker_id === null) {
    $db->query("UPDATE meetup_slots SET speaker_id = $userId WHERE id = 1");
}
```

**Проблема:** два пользователя одновременно читают `speaker_id IS NULL`. Оба выполняют `UPDATE`. Второй пользователь перезаписывает первого (потерянное обновление).

### Решения:
1.  **Пессимистическая блокировка (`SELECT FOR UPDATE`)**:
    Блокирует строку до завершения транзакции. Другие транзакции будут ждать на этапе `SELECT`.
    *   **`NOWAIT`**: если вы хотите, чтобы запрос упал сразу, а не ждал освобождения блокировки:
        `SELECT * FROM meetup_slots WHERE id = 1 FOR UPDATE NOWAIT;`
    *   **`SKIP LOCKED`**: идеально для очередей. Пропускает строки, уже заблокированные другими:
        `SELECT * FROM jobs WHERE status = 'pending' LIMIT 1 FOR UPDATE SKIP LOCKED;`

2.  **Атомарные обновления (рекомендуется)**:
    Объедините проверку и обновление в одном SQL-запросе:
    ```sql
    UPDATE meetup_slots 
    SET speaker_id = :user_id 
    WHERE id = :id AND speaker_id IS NULL;
    ```
    Всегда проверяйте количество **Rows Affected**. Если оно равно `0`, значит условие уже не выполняется и операция не удалась.

## 2. Кейс №2: Идемпотентность (обработка повторов)

### Сценарий
API для добавления посетителя. Из-за сетевых сбоев или гарантии доставки "at-least-once" (Kafka) один и тот же запрос может прийти несколько раз.

**Проблема:** пользователь может быть добавлен несколько раз, занимая лишние места.

### Решение:
1.  **Unique Constraints**: создайте уникальный индекс на пару `(meetup_id, user_id)`.
2.  **Обработка ошибок**:
    Перехватывайте ошибку `Duplicate Key`. Если произошел повтор, база заблокирует вторую вставку. Обработайте это как успех (или состояние «уже сделано»), чтобы клиент получил консистентный ответ.

## 3. Кейс №3: Управление лимитами (агрегаты)

### Сценарий
Гарантия того, что количество участников не превысит лимит (например, 100 мест).

**Наивная реализация:**
```sql
SELECT COUNT(*) FROM visitors WHERE meetup_id = 1; -- Возвращает 99
-- ... если count < limit, то ...
INSERT INTO visitors (meetup_id, user_id) VALUES (1, 101);
```

**Проблема:** параллельные транзакции могут обе увидеть `99` и обе выполнить вставку, в итоге участников станет `101`.

### Решения:
1.  **Блокировка родительской строки**:
    `SELECT * FROM meetups WHERE id = 1 FOR UPDATE;`
    Это выстраивает все регистрации на *конкретный митап* в очередь.
2.  **Денормализация + CHECK Constraint**:
    Добавьте колонку `booked_seats` в таблицу `meetups` и ограничение `CHECK (booked_seats <= total_seats)`.
    Обновляйте счетчик в той же транзакции, что и вставку:
    ```sql
    UPDATE meetups SET booked_seats = booked_seats + 1 WHERE id = 1;
    ```
    База данных обеспечит выполнение `CHECK` для всех транзакций.

## 4. Распределенные блокировки (Redis vs SQL)

### Когда использовать Redis:
-   **Rate Limiting**: предотвращение шквала запросов, перегружающих БД.
-   **Circuit Breakers**: временная блокировка доступа к проблемным ресурсам.
-   **Синхронизация задач**: гарантия того, что Cron-задача запустится один раз на весь кластер.

### Когда избегать Redis для консистентности данных:
Если требование консистентности относится строго к БД, используйте функции SQL.
-   **Нет кросс-системной атомарности**: вы не можете атомарно закоммитить ключ в Redis и строку в SQL.
-   **Риски TTL**: блокировка в Redis может истечь, пока транзакция в БД еще выполняется.

## 5. Глубокое погружение: MVCC (Multi-Version Concurrency Control)

Базы данных управляют конкурентностью, сохраняя несколько версий строки. Это позволяет одной транзакции читать данные, пока другая их обновляет, без взаимных блокировок.

### PostgreSQL
Postgres сохраняет все версии строк прямо в основной таблице (heap). У каждой строки есть скрытые системные колонки:
- **`xmin`**: ID транзакции, которая вставила строку.
- **`xmax`**: ID транзакции, которая удалила или обновила строку.
- **`ctid`**: физический адрес версии строки на диске.

### MySQL (InnoDB)
MySQL хранит только последнюю версию строки в таблице, а исторические данные выносит в **Undo Log**:
- **`DB_TRX_ID`**: ID последней транзакции, вставившей или обновившей строку.
- **`DB_ROLL_PTR`**: указатель на предыдущую версию в Undo Log (через который можно «отмотать» изменения).
- **`DB_ROW_ID`**: скрытый ID строки (используется, если у таблицы нет первичного ключа).

### Режимы блокировок строк (PostgreSQL)
Postgres предоставляет тонкий контроль над блокировками:
- **`FOR UPDATE`**: полная эксклюзивная блокировка. Запрещает любые другие блокировки.
- **`FOR NO KEY UPDATE`**: запрещает `FOR UPDATE` и другие `FOR NO KEY UPDATE`, но разрешает `FOR KEY SHARE`. Используется при обновлении колонок, не входящих в уникальные индексы.
- **`FOR SHARE`**: разделяемая блокировка. Разрешает другим читать (`FOR SHARE`), но запрещает изменение (`FOR UPDATE`).
- **`FOR KEY SHARE`**: самая слабая блокировка. Разрешает `FOR NO KEY UPDATE`.

### Advisory Locks (Рекомендательные блокировки)
Это логические блокировки, которые база не применяет автоматически — ваше приложение должно само их проверять.
- **Уровень сессии**: `SELECT pg_advisory_lock(123);` действует до конца сессии или вызова `pg_advisory_unlock`.
- **Уровень транзакции**: `SELECT pg_advisory_xact_lock(123);` автоматически освобождается в конце текущей транзакции (`COMMIT` или `ROLLBACK`). Это крайне полезно для синхронизации высокоуровневой логики приложения внутри БД-транзакций.

## 6. Чек-лист для разработчика

- [ ] **Rows Affected**: всегда проверяйте `rows_affected` после `UPDATE` или `DELETE`.
- [ ] **Ошибки SQL**: обрабатывайте нарушение констрейнтов, Deadlocks и ошибки сериализации.
- [ ] **Ожидание**: используйте `NOWAIT` для интерактивных действий в UI и `SKIP LOCKED` для фоновых воркеров.
- [ ] **Короткие транзакции**: держите блокировки минимально необходимое время.
- [ ] **Идемпотентность**: убедитесь, что повторы безопасны и возвращают тот же финальный статус.

---

## 7. Чек-лист для System Design

1.  **Конкурентность**: есть ли одновременный доступ к одной сущности от разных пользователей?
2.  **Фрод/Абуз**: может ли пользователь попытаться «обмануть» систему с нескольких устройств?
3.  **Уровень консистентности**: нужна ли **Strong Consistency** или допустима **Eventual Consistency**?
4.  **Идемпотентность**: является ли API идемпотентным?
5.  **Режимы отказа**: что произойдет, если связь оборвется между коммитом в БД и ответом клиенту?
